## 01 — How I Think About Engineering

I didn't start out drawn to "coding" as an abstract skill — I got hooked on the moment a product actually *works end-to-end*: a user adds something to a cart, it survives a page refresh, and the database quietly does its job in the background. That's the part of software engineering I care about most.

So instead of picking a single lane, I've deliberately worked across the stack — designing the interface in Figma, wiring up the React state, writing the REST endpoints, and normalizing the database schema underneath it. Not because I'm trying to be a generalist for its own sake, but because I think **the best full-stack engineers understand how a decision in one layer quietly becomes a bug (or a win) in another.**

I'm still early in my journey — a third-year CSE undergraduate — but every project below was built the way I'd want to build in a real engineering team: version-controlled, structured, and iterated on based on feedback.

Every project here answers the same question differently: **"What does this system need to remember, and how do I model that honestly?"**
